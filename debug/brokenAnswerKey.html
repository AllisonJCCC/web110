Detailed Bug Breakdown

01 Unclosed <section class="hero"> Validator

What's wrong

The hero <section> is opened but never closed. This means the pets grid, contact section, and footer are all technically nested inside the hero — breaking document structure and causing unexpected background colors and spacing to bleed into the rest of the page.

Broken code

<section class="hero">
  <h2>Give a Pet a Second Chance</h2>
  <p>Every animal at Riverside...</p>
                               <!-- no closing tag! -->
<h2 class="section-title">Meet Our Animals</h2>

Fixed code

<section class="hero">
  <h2>Give a Pet a Second Chance</h2>
  <p>Every animal at Riverside...</p>
</section>

<h2 class="section-title">Meet Our Animals</h2>
W3C Validator → flags unclosed element or unexpected end tag
02 Missing alt attributes on all four <img> tags Validator

What's wrong

Every <img> tag on the page is missing a required alt attribute. This fails W3C validation and is an accessibility violation — screen readers used by visually impaired users have nothing to announce for these images.

Broken code

<img src="cat.webp" width="120" height="120">
<img src="dog.webp" width="120" height="120">
<img src="bunny.webp" width="120" height="120">
<img src="hamster.webp" width="120" height="120">

Fixed code

<img src="cat.webp" width="120" height="120" alt="Luna the tabby cat">
<img src="dog.webp" width="120" height="120" alt="Biscuit the beagle mix">
<img src="bunny.webp" width="120" height="120" alt="Clover the Holland Lop rabbit">
<img src="hamster.webp" width="120" height="120" alt="Peanut the hamster">
W3C Validator → "An img element must have an alt attribute" (reported four times)
03 <em> wrapping block-level <h3> DevTools

What's wrong

<em> is an inline element. It cannot wrap a block-level element like <h3>. The browser silently restructures the DOM to fix this — but the result is that the italic styling may not apply as expected, and the document structure is invalid. The W3C validator will not catch this in HTML5 mode.

How to see it

Open DevTools → Elements panel. You will see the browser has restructured the markup — the <em> has been moved inside the <h3>, not around it.

Broken code

<em>
  <h3>Peanut</h3>
</em>

Fixed code

<h3><em>Peanut</em></h3>
DevTools → Elements panel shows the browser rewrote the DOM structure
04 <p> nested inside <p> DevTools

What's wrong

The Peanut card has a <p class="pet-card"> wrapping the entire card content, and then another <p> for the description text inside it. A <p> tag cannot contain another <p> — paragraphs are block elements and cannot nest. The browser will auto-close the outer <p> early, breaking the card layout.

How to see it

Open DevTools → Elements panel. The Peanut card will look structurally different from the other three cards. The outer <p> will have been auto-closed by the browser before the inner one opens.

Broken code

<p class="pet-card">               <!-- outer p opens -->
  <img src="hamster.webp"...>
  <h3>Peanut</h3>
  <p>6-month-old hamster...</p>   <!-- inner p — invalid! -->
  <a href="#">Adopt Me</a>
</p>

Fixed code

<article class="pet-card">
  <img src="hamster.webp"...>
  <h3><em>Peanut</em></h3>
  <p>6-month-old hamster...</p>
  <a href="#">Adopt Me</a>
</article>
DevTools → Elements panel shows the Peanut card has different DOM structure than the other three cards
05 Missing </body> closing tag Validator

What's wrong

The file ends with </html> but is missing the </body> tag before it. Browsers handle this gracefully through error recovery, but it is invalid HTML and will be flagged by the validator. In larger projects, a missing </body> can cause scripts placed before it to behave unexpectedly.

Broken code

  <footer>...</footer>

</html>

Fixed code

  <footer>...</footer>

</body>
</html>
W3C Validator → "End of file seen when there were open elements" or missing end tag warning
06 Nav hover: text becomes invisible CSS Visual

What's wrong

In the nav a:hover rule, both color (the text) and background-color are set to the same value — #2c5440. When a user hovers over a nav link, the text disappears completely because it is the same color as the background. No validator or DevTools error will appear; this is a pure visual logic bug.

How to see it

Hover your mouse over any navigation link. The text vanishes. Open DevTools → Elements → force the :hover state in the Styles panel to inspect both values side by side.

Broken code

nav a:hover {
  color: #2c5440;            /* same value as background! */
  background-color: #2c5440;
  padding: 0.2rem 0.5rem;
}

Fixed code

nav a:hover {
  color: #ffffff;            /* white text — visible on dark background */
  background-color: #2c5440;
  padding: 0.2rem 0.5rem;
}
CSS Visual bug — hover over a nav link and watch the text disappear
07 Hero paragraph not centered: margin: auto 0 CSS Visual

What's wrong

CSS shorthand margin: auto 0 means top and bottom = auto, left and right = 0. To horizontally center a block element with a fixed width, you need margin: 0 auto, which sets left and right margins to auto — pushing the element to the middle. The hero paragraph appears left-aligned instead of centered despite having max-width: 600px set.

How to see it

The hero text sits against the left edge of the page instead of being centered. Open DevTools → Box Model — you will see the left and right margins are both 0.

Broken code

.hero p {
  max-width: 600px;
  margin: auto 0;    /* top/bottom = auto — does NOT center horizontally */
}

Fixed code

.hero p {
  max-width: 600px;
  margin: 0 auto;    /* left/right = auto — centers the block element */
}
CSS Visual bug — hero paragraph is left-aligned. DevTools Box Model confirms left/right margins are 0.
